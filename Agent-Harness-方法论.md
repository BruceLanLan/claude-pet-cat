# Agent Harness 开发方法论

> 从 Claude Code 源码、设计指南、综合分析中提炼的通用框架
> 适用于：AI Agent、工具系统、上下文管理、记忆工程

---

## 核心理念

```
能力 ≠ 模型大小
壁垒 = Harness(约束) + Context(上下文) + Memory(记忆) + Safety(安全)
```

**工程化约束比追求模型能力更重要。**

## 第一维度：运行时 (Runtime)

### 1.1 必选模式

#### A. PDAOL 循环（控制平面）

```
Perceive → Decide → Act → Observe → Loop

实现要点：
- 不要设计"一问一答"，要设计"控制循环"
- 每次 Action 后必须有 Observe 阶段
- Loop 要有退出条件（成功/失败/预算耗尽）
```

#### B. 不可变消息历史

```
原则：消息只追加，从不修改

✅ 追加 → 压缩（生成摘要） → 替换历史
❌ 修改 → 覆盖 → 丢失状态
```

#### C. Token 预算双维度

```
维度1: Token 数量（防上下文溢出）
维度2: USD 成本（防账单超支）

触发时机：
  85% → 警告
  95% → 压缩
  100% → 停止 + 优雅退出
```

### 1.2 推荐模式

#### D. 流式事件架构

```
事件类型（必须定义）：
  - message_start     → 开始
  - tool_match        → 工具命中
  - permission_denial → 权限拒绝
  - message_delta     → 内容片段
  - message_stop      → 结束

实现：Generator yield，天然背压
```

#### E. 二元反馈自愈（Binary Feedback）

```
同一请求执行两次 → 比对结果
  结果一致 → 继续
  结果不一致 → 触发反思/重试

适用场景：代码执行、工具调用、外部 API
```

### 1.3 可选模式

#### F. 状态机模式

```
状态流转：
  pending → running → completed/failed/killed
                              ↑ 终态不可逆

Task ID 生成：
  前缀(b=bash/a=agent/r=remote) + 8位 base-36 随机
  → 防枚举攻击
```

## 第二维度：工具系统 (Tools)

### 2.1 原子化设计

```
单一职责原则：
  ❌ 一个大而全的工具做所有事
  ✅ 多个小工具可组合

工具接口四要素：
  1. name        → 工具名（为 AI 编写，非用户）
  2. description → 描述（影响 AI 选择质量）
  3. input_schema → 输入格式
  4. execute()   → 异步执行函数
```

### 2.2 工具发现机制

#### 方案 A：延迟加载（推荐用于大规模工具）

```
问题：50个工具定义 = 77K tokens

解法：
  1. 大部分工具标记 defer_loading
  2. 只保留一个轻量的 Tool Search Tool
  3. 按需动态拉取 3-5 个最相关工具
  4. 初始开销锐减 85%

适用：MCP 服务器、插件系统
```

#### 方案 B：快照镜像（适用于移植/重写）

```
结构：
  reference_data/tools_snapshot.json
      ↓ @lru_cache 单例加载
  Python 层纯展示

适用：干净室重写、接口迁移
```

### 2.3 工具路由

```
Prompt Token 路由算法：
  1. Tokenize: "file-edit" → ['file', 'edit']
  2. 打分: token 在 name/description/responsibility 出现 = 1分
  3. 排序: 降序取 top-N
  4. 填充: 每类（command/tool）各取最优

优化：source_hint 字段做分类（builtin/plugin/skill）
```

## 第三维度：上下文 (Context)

### 3.1 缓存友好的 Prompt 结构

```
┌──────────────────────────────────────────────┐
│  A. 稳定缓存区（开头，最大化 KV cache 命中） │
│    - 系统指令                                 │
│    - 工具定义                                 │
│    - 项目规范 (CLAUDE.md)                     │
├──────────────────────────────────────────────┤
│  B. 动态增量区（每次变化）                   │
│    - git status（截断 2000 chars）            │
│    - 用户上下文                               │
│    - 近期记忆指针                             │
├──────────────────────────────────────────────┤
│  C. 对话历史区（60% token 预算）             │
│    - 消息列表                                │
│    - 工具执行结果                             │
├──────────────────────────────────────────────┤
│  D. 输出预算区（10% token）                  │
│    - 保留空间给回复                           │
└──────────────────────────────────────────────┘
```

### 3.2 上下文注入优先级

```
优先级 1: user-append prompts（用户追加）
优先级 2: custom prompts（自定义）
优先级 3: CLAUDE.md / 项目上下文
优先级 4: default instructions（默认）
```

### 3.3 CLAUDE.md 规范

```
文件位置：项目根目录 /.claude/CLAUDE.md

结构模板：
  # 项目名称
  技术栈: [列表]
  目录结构: [描述]
  代码规范: [要点]
  常用命令: [快捷命令]
  重要约定: [团队规范]

限制：
  - 单文件 < 2000 characters
  - 支持 @ 语法引用外部文件
  - 支持多级：global → project → subdirectory
```

## 第四维度：记忆系统 (Memory)

### 4.1 记忆分层架构

```
┌─────────────────────────────────────────┐
│  L0: MEMORY.md（全局轻量索引）           │
│      - 200行/25KB 以内                   │
│      - 指向具体主题文件的指针             │
│      - 常驻内存，快速加载                 │
├─────────────────────────────────────────┤
│  L1: 短期记忆（会话/日/周）              │
│      - 对话历史                          │
│      - 每日笔记 memory/YYYY-MM-DD.md     │
│      - 任务结果                          │
├─────────────────────────────────────────┤
│  L2: 长期记忆（信念/决策/知识图谱）       │
│      - SQLite 数据库                      │
│      - 置信度 + 证据追踪                  │
│      - 向量语义搜索（可选）               │
├─────────────────────────────────────────┤
│  L3: 归档记忆（冷数据）                  │
│      - ZIP 快照导出                       │
│      - 可跨环境热插拔                     │
└─────────────────────────────────────────┘
```

### 4.2 记忆类型定义

| 类型 | 内容 | 存储 | 加载策略 |
|------|------|------|----------|
| **user** | 用户偏好、习惯 | L1 | 主会话加载 |
| **feedback** | 行为纠正、成功/失败模式 | L2 | 相关任务时加载 |
| **project** | 项目决策、架构选择 | L0+L2 | 项目上下文时加载 |
| **reference** | 外部资源、文档链接 | L2 | 按需加载 |

### 4.3 记忆写入原则

```
✅ 保存：
  - 非显而易见的信息（团队约定、历史决策）
  - "为什么"而非"做了什么"
  - 带有时间戳（防陈旧）

❌ 不保存：
  - 可推导的信息（代码结构、文件路径）
  - 纯过程细节
  - 已被替代的决策
```

### 4.4 记忆相关性加载

```
旧方案：全量加载
新方案：相关性搜索

步骤：
  1. 解析当前任务关键词
  2. 计算与信念的语义/字面相关性
  3. 只返回 top-N 相关记忆
  4. 访问后自动强化（boost +0.05，上限 80%）
```

### 4.5 AutoDream 记忆整理（高级）

```
触发条件：
  - 离线空闲
  - ≥24h 未整理
  - ≥5 个新会话

四阶段后台整理：
  1. Orient（定向）→ 确定当前项目状态
  2. Gather（收集）→ 收集相关记忆片段
  3. Consolidate（合并）→ 纠错消重
  4. Prune（裁剪）→ 删除过时/矛盾信息

输出：
  - 相对日期 → 绝对日期
  - 矛盾标记 → 冲突解决
  - 长篇 → 精简摘要
```

## 第五维度：安全 (Safety)

### 5.1 五层权限架构

```
Layer 1: Session Mode（全局基调）
  - default      → 危险操作询问
  - acceptEdits  → 自动接受编辑
  - bypassPermissions → 跳过所有检查（⚠️危险）
  - plan         → 只生成计划不执行

Layer 2: Tool Whitelist/Blacklist
  - settings.json 或 CLAUDE.md 配置

Layer 3: Tool-Level Permissions
  - 读工具 → 自动放行
  - 写/执行工具 → 需确认

Layer 4: Operation-Level Permissions
  - 同一工具不同风险（rm -rf vs git status）

Layer 5: Path/Command-Level Permissions
  - 特定目录/文件/命令的精确控制
```

### 5.2 自适应权限（YOLO 模式）

```
原理：ML 分类器分析上下文连贯性，自动决策是否跳过审批

规则简化版：
  - 读操作 → 上下文连贯则自动放行
  - 写操作 → 首次需要确认，后续学习
  - Shell 高危（rm -rf）→ 物理隔离/强制确认

实现：置信度阈值 + 历史行为学习
```

### 5.3 信任门控

```
启动序列中的 Trust Gate：

Stage 1: prefetch（轻量检查）
Stage 2: warning + environment guards
Stage 3: CLI parser + pre-action trust gate
Stage 4: setup() + parallel load（重量级）
Stage 5: deferred init（trust 验证后才执行）

重量级操作延迟到 trust 确认后：
  - 插件初始化
  - MCP 预取
  - 会话 hooks
  - 外部网络请求
```

## 第六维度：可进化 (Evolution)

### 6.1 短反馈循环

```
触发：每次任务完成

流程：
  成功 → 强化成功模式（confidence +）
  部分成功 → 分析可改进点
  失败 → 记录教训 + 建议备选

存储：最近 100 条
```

### 6.2 长反馈循环

```
触发：周期性（周/月）

分析：
  - 按任务类型统计成功率
  - 识别改进趋势（improving/declining）
  - 战略性调整建议

策略调整：
  - expand: 成功率 > 80%，扩大范围
  - consolidate: 成功率 < 50%，收缩聚焦
  - investigate: 下滑类型深查
  - amplify: 上升类型增加频率
```

### 6.3 A/B 测试框架

```
启动测试：
  start_ab_test(test_id, task_type, variant_a, variant_b)
  min_samples: 10

记录结果：
  record_ab_result(test_id, variant, outcome)

评估：
  胜出条件：某变体成功率 > 另一变体 + 10%
  样本不足：继续积累
  结果不明：宣布 inconclusive
```

### 6.4 渐进复杂度模型

```
Level 1: 自然语言
  → 新用户，简单对话

Level 2: 配置驱动
  → power user，settings.json / CLAUDE.md

Level 3: 扩展集成
  → MCP 服务器 / Skills

Level 4: 多代理编排
  → AgentTeam / Coordinator 模式
```
